# Исправление бага «чёрный экран при перемещении карты»

## Корневая причина

Картинка карты **6336×2688 px (~10 MB)**. При `transform: scale(2)` это даёт CSS-слой **12672×5376** логических пикселей, а на Retina-дисплее с DPR 2× — фактически **25344×10752** пиксельная текстура. Это превышает максимальный размер GPU-текстуры в Chrome/Safari/Firefox (обычно 4096–16384 px в зависимости от GPU).

Браузер не может растеризовать такой огромный композитный слой и в момент панорамирования слой пропадает — видно тёмный градиент `body`. Это и есть «чёрный экран». Атрибут `will-change: transform` на `.map-world` усугубляет: он принудительно создаёт композитный слой именно из этой картинки.

## Что нужно поменять (3 файла)

### 1. Заменить картинку

В архиве рядом с этим документом лежит **`map-base.jpg`** — та же ваша карта, но 3168×1344 (1.1 MB вместо 10 MB). Визуально неотличимо: даже при максимальном зуме вы рендерите её на 6336×2688 эффективных пикселей, что и есть исходный размер.

Положите этот файл вместо вашего `map-base.jpg`.

### 2. В `data.jsx` поменять размеры карты

Найдите в конце файла:

```js
window.MAP_W = 6336;
window.MAP_H = 2688;
```

Замените на:

```js
window.MAP_W = 3168;
window.MAP_H = 1344;
```

(Координаты пинов в процентах не трогаем — они привязаны к размеру в %, не в пикселях.)

### 3. Заменить `app.jsx` целиком

Файл `app.jsx` рядом с этим документом — это полная замена с четырьмя исправлениями:

**Fix 1 — drag больше не блокируется на пинах и hit-областях.** В оригинале `onPointerDown` рано выходил, если клик пришёлся на `.pin` / `.river-hit` / `.coffee-hit`. Это не давало драгать, если стартовал тач с этих элементов (а на маленьких экранах они занимают пол-карты). Теперь drag всегда стартует, а сами пины/hit-области не открывают попап, если drag уже сдвинулся (`drag.current.moved`).

**Fix 2 — wheel-зум переехал на нативный listener с `{ passive: false }`.** В современном React `onWheel` пассивный, поэтому `e.preventDefault()` молча игнорируется и в консоли спамит warning. Native listener через `useEffect` работает.

**Fix 3 — анимация `panToStop` теперь клампит каждый кадр.** Раньше интерполировались сырые значения, и на 1–2 кадрах позиция могла выйти за границы.

**Fix 4 — defensive guards.** `safeView()` ловит `NaN` / `Infinity` / отрицательный scale и сбрасывает на fit-to-screen. `clamp` заодно докручивает scale до `minScale` если кто-то его опустил ниже. На draggable изменён `translate(...)` → `translate3d(..., 0)` — это форсит хардверное ускорение и стабильнее под нагрузкой.

### 4. В CSS убрать `will-change` с map-world

В вашем `Magic School Map.html`, секция `<style>`, найдите:

```css
.map-world{
  position:absolute;left:0;top:0;
  transform-origin:0 0;
  will-change:transform;
}
```

Уберите строку `will-change:transform;`. Замените на:

```css
.map-world{
  position:absolute;left:0;top:0;
  transform-origin:0 0;
  /* will-change убрано — пусть браузер сам решает, когда композитить */
  backface-visibility:hidden;
}
```

`backface-visibility:hidden` оставляет лёгкий хинт композитору, но не форсит создание гигантского слоя.

## Проверка

После этих изменений:
- Картинка грузится в ~10 раз быстрее (1.1 MB)
- При зуме до 2× эффективная текстура — 6336×2688, что укладывается в лимиты всех современных GPU
- Drag работает с любой точки карты, включая поверх пинов
- Wheel-зум работает без warning-ов в консоли
- Случайный клик после drag не открывает пин

## Если бы строили с нуля

Я бы взял библиотеку **`react-zoom-pan-pinch`** — она решает все эти проблемы из коробки (bounded pan, корректный wheel-listener, поддержка pinch-zoom на тач-устройствах, suppression клика после drag, плавная инерция). Но для текущего кода точечные патчи дешевле, чем переписывать.
